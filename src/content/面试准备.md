
文档相关 bug

- 问题是Blocksuite 存在全局样式注入
	- 来源于包内 CSS
- 为了解决这一点使用 iframe隔离
	- 设置了一个路由，内部渲染编辑器组件，作为 iframe 的 src
	- 包装一个 BlockSuiteDescriptionEditor组件，内部渲染了路由为 src 的 iframe

- docId 做动态路由进行 src 切换导致iframe 每次会重新挂载，因为 iframe相当于一个子浏览器环境，加载的过程会加载 CSS 等导致白屏闪烁
	- 解决方案是使用useRef包装这个 initParam，然后把 useMemo 的依赖改为 initParam 的ref容器，这样 src 就不会更新了
	- 这样，我们既可以保存改变后的信息，也不会触发 iframe 的reload
	- 依赖保存后的信息，由主线程给 iframe 内部postMessageFrame来更新内部的信息

- 不 reload 之后，shadow dom方案的残留，将 CSS 卸载了
	- 让 AI 分析关于 CSS 的时序逻辑，将冗余的隔离和守卫代码删去

- 状态残留，切换文章时，旧文章的状态会先同步到侧边栏，再恢复正常，出现了闪烁行为
	1. Host 发送 sync-params -> Frame 更新 docId。
	2. Runtime 先触发 tcHeaderState effect（旧 state + 新 docId） -> 外部被错误覆盖。
	3. 新 doc 的 header 初始化/订阅完成 -> tcHeaderState 变新 -> 外部再被修正。

	- 解决方案是，切换时先清空状态，保证旧状态再切换时不会重新写入
	- 其次做版本校验，用一个 ref 标记最新的状态值，与闭包保存的异步值做比较，只有他们相等时才允许写入状态

图像批量裁剪

- 问题是用户需要针对立绘渲染，对图片进行批量的二次裁剪，显然这个过程串行是比较低效的，前端这里主要的耗时就是针对图片的 IO 计算，所以使用 webworker 并行化是自然的想法
- 实现方案是，创建一个 worker 池类，创建等待队列，可执行队列和 worker 队列，管理调度任务
- 大致过程是，
	- worker 处理任务，完成后发送消息给主线程
	- 主线程将这个 worker 加入可执行worker队列
	- 检查等待队列有任务，则使用拿出可执行队列里的 worker，拿出等待队列的任务，发送任务给 worker 线程
	- 如果没有可执行 worker，任务就进入等待队列中
