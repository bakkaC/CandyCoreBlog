
### 图像批量裁剪

- 问题是用户需要针对立绘渲染，对图片进行批量的二次裁剪，显然这个过程串行是比较低效的，前端这里主要的耗时就是针对图片的 IO 计算，所以使用 webworker 并行化是自然的想法
- 实现方案是，创建一个 worker 池类，创建等待队列，可执行队列和 worker 队列，管理调度任务
- 大致过程是，
	- worker 处理任务，完成后发送消息给主线程
	- 主线程将这个 worker 加入可执行worker队列
	- 检查等待队列有任务，则使用拿出可执行队列里的 worker，拿出等待队列的任务，发送任务给 worker 线程
	- 如果没有可执行 worker，任务就进入等待队列中

### 图像立绘预览

- 问题在于立绘列表是一个需要高频切换，并要求预览渲染内容的组件，而渲染预览的立绘是一个需要计算的异步操作，这会导致切换操作下出现渲染的时序问题，具体问题如下
	- 计算完成前，会出现旧立绘残留，包括立绘像素和 transition 参数的计算
	- 连续切换时，最后计算完成的可能不是当前选中立绘

- 状态残留的问题，解决方案是，切换发生时先清空旧状态，然后等待新状态就续再渲染
	- 在 useLayoutEffect，即 Mutation 完成后就清空旧立绘，不会绘制旧立绘到浏览器上
	- 新图片 onload 后再设置裁剪框参数，在绘制前应该提前加载 CSS transition 的参数，就是在裁剪的 onImageLoad 里加一个可选执行的函数 onImageExtend 用于加载 transition 参数
		- 在这个场景里它会 setDisplayTransform（SpriteCropper）。
		- 如果马上 drawInitialPreview，因为 drawInitialPreview 是异步的，可能出现“新 canvas 画面 + 旧 transform/旧布局”的一帧错配，用户会看到闪一下。
		- 用 requestAnimationFrame(drawInitialPreview) 把首次绘制放到下一帧，给这些状态提交和CSS transition留出时间
	-  避免旧帧出现的清空策略和避免闪烁的设置状态会在首次挂载冲突
		- 从无到有的状态会触发清空，导致首次加载的画布被清空
		- 所以解决方式是跳过首次挂载的 useLayoutEffect，利用一个持久存在的 ref值记录是否是首屏状态，利用这个变量在 useLayoutEffect 内部就能提前跳过
- 快速切换的问题，解决方案是设置一个 currentkey存储 URL，然后用 ref包裹一个 latestkey，前者会在多次执行里闭包保存状态，而 latest 始终能拿到当前的状态
	- 闭包捕获的是：countRef 这个对象，不是 `countRef.current` 的值。
	- 两者校验决定是否准备好了，准备好了才渲染


### 文档相关 bug

- 问题是Blocksuite 存在全局样式注入
	- 来源于包内 CSS
- 为了解决这一点使用 iframe隔离
	- 设置了一个路由，内部渲染编辑器组件，作为 iframe 的 src
	- 包装一个 BlockSuiteDescriptionEditor组件，内部渲染了路由为 src 的 iframe

- docId 做动态路由进行 src 切换导致iframe 每次会重新挂载，因为 iframe相当于一个子浏览器环境，加载的过程会加载 CSS 等导致白屏闪烁
	- 解决方案是使用useRef包装这个 initParam，然后把 useMemo 的依赖改为 initParam 的ref容器，这样 src 就不会更新了
	- 这样，我们既可以保存改变后的信息，也不会触发 iframe 的reload
	- 依赖保存后的信息，由主线程给 iframe 内部postMessageFrame来更新内部的信息

- 不 reload 之后，shadow dom方案的残留，将 CSS 卸载了
	- 让 AI 分析关于 CSS 的时序逻辑，将冗余的隔离和守卫代码删去

- 状态残留，切换文章时，旧文章的状态会先同步到侧边栏，再恢复正常，出现了闪烁行为
	1. Host 发送 sync-params -> Frame 更新 docId。
	2. Runtime 先触发 tcHeaderState effect（旧 state + 新 docId） -> 外部被错误覆盖。
	3. 新 doc 的 header 初始化/订阅完成 -> tcHeaderState 变新 -> 外部再被修正。

	- 解决方案是，切换时先清空状态，保证旧状态再切换时不会重新写入
	- 其次做版本校验，用一个 ref 标记最新的状态值，与闭包保存的异步值做比较，只有他们相等时才允许写入状态


### 双层结构与乐观更新

- 直接修改缓存的问题是类型安全和并发修改的问题

- 并发修改问题
	- 乐观更新本质是手动修改 cache
	- 但后台的 query 或其他 mutation 也会写入 cache
	- 如果这些操作没有正确序列化，会覆盖彼此的结果
- 解决方案是，在 onMutate 里使用 cancelQueries 和 onSettled里用 isMutating 判断是否有 hooks 正在执行，再 invalidqueries，如果要解决类型安全的问题，需要再设置 queryoption，进行传入，这里就是重复繁琐的配置项的问题
- 但是直接加一层本地的 role 状态用于渲染，这不就解决了类型安全和缓存冲突的问题了吗，为 state 设置类型是符合直觉的
- 本地状态的问题是两个状态源带来的一致性问题，这一点是在每次server 操作后根据结果进行手动同步
