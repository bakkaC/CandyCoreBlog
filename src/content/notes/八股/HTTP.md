## HTTP版本


HTTP 发展主要有 **HTTP/1.0、HTTP/1.1、HTTP/2 和 HTTP/3**。

最早的 **HTTP/1.0** 用的是**短连接**，也就是每发一个请求都要建立一次 TCP 连接，请求完成就关闭。如果一个网页有很多资源，比如图片、CSS、JS，就要反复建立连接，效率比较低。

后来 **HTTP/1.1** 做了优化，最重要的是引入了 **长连接（Keep-Alive）**，一个 TCP 连接可以发送多个请求，这样减少了频繁建立连接的开销。另外还支持 **管道化请求**，可以一次发送多个请求，不过因为会有**队头阻塞**问题，所以实际用得不多。

再后来是 **HTTP/2**，它主要解决 HTTP/1.1 的性能问题。HTTP/2 把协议改成了 **二进制格式**，提高了解析效率，并且支持 **多路复用**，也就是在一个 TCP 连接里可以同时处理多个请求，不需要排队等待。另外还引入了 **头部压缩** 和 **服务器推送**。

最后是 **HTTP/3**，HTTP/2 虽然解决了 HTTP 层的阻塞问题，但底层还是 TCP，如果 TCP 丢包还是会有 **队头阻塞**。所以 HTTP/3 直接改成基于 **QUIC 协议（UDP）**，这样可以更快建立连接，也能减少丢包带来的影响，性能在移动网络环境下会更好。

## HTTP 报文结构

HTTP 报文主要分成 **请求报文** 和 **响应报文**，两者的结构其实很类似，基本都由三部分组成：**起始行、头部字段、消息体**。

先说 **HTTP 请求报文**。

请求报文一般包含四部分：**请求行、请求头、空行、请求体**。

请求行主要包含三个信息：**请求方法、请求的 URL、HTTP 版本**。  
比如：

GET /index.html HTTP/1.1

接下来是 **请求头**，也就是一些键值对，用来描述客户端的信息，比如：

Host: www.example.com  
User-Agent: Chrome  
Accept: text/html  
Cookie: xxx

请求头后面会有一个 **空行**，这个空行的作用是用来分隔 **头部和消息体**。

最后是 **请求体**，也就是请求的数据，一般在 **POST、PUT** 这种请求里会有，比如提交表单或者 JSON 数据。

---

然后是 **HTTP 响应报文**。

响应报文结构也差不多，包括：**状态行、响应头、空行、响应体**。

**状态行**包含三个信息：**HTTP版本、状态码、状态码描述**。  
比如：

HTTP/1.1 200 OK

接下来是 **响应头**，用于描述服务器返回的信息，例如：

Content-Type: text/html  
Content-Length: 1024  
Set-Cookie: sessionid=xxx

然后同样会有一个 **空行**，用于分隔头部和内容。

最后是 **响应体**，也就是服务器返回的具体数据，比如 HTML 页面、JSON 数据、图片等。

## 常见方法和状态码

| 方法          | 作用              |
| ----------- | --------------- |
| **GET**     | 获取资源（最常见）       |
| **POST**    | 提交数据（如表单）       |
| **PUT**     | 替换资源            |
| **PATCH**   | 部分更新资源          |
| **DELETE**  | 删除资源            |
| **HEAD**    | 只请求头，不取内容（用于探测） |
| **OPTIONS** | 查询服务器支持哪些方法     |

1. 语义上 GET 用来查数据，POST 用于修改数据
2. GET 请求是幂等的，多次调用不会有副作用，POST请求是非幂等的
3. 浏览器对 URL 有长度限制，POST 理论上没限制，但服务端通常会加限制
4. GET 会被浏览器和 CDN 缓存，POST不会

两者都不安全，安全性依赖 HTTPS，GET 也可以传请求体，用来改数据，POST 也可以用 URL 传参，用来查数据

全用 POST 可以把约定请求数据方式，减小沟通成本，
但是基建的缓存和重试一些策略会失效，GET 请求 URL和参数比较直观，POST 请求还需要构造请求体无形之中会增加排查问题的成本

| 状态码                           | 含义    |
| ----------------------------- | ----- |
| **200 OK**                    | 请求成功  |
| **301 Moved Permanently**     | 永久重定向 |
| **302 Found**                 | 临时重定向 |
| **304 Not Modified**          | 使用缓存  |
| **400 Bad Request**           | 请求错误  |
| **401 Unauthorized**          | 需要认证  |
| **403 Forbidden**             | 无权限访问 |
| **404 Not Found**             | 找不到资源 |
| **500 Internal Server Error** | 服务器错误 |
| **502 Bad Gateway**           | 网关错误  |
| **504 Gateway Timeout**       | 网关超时  |

## HTTP缓存

HTTP 的缓存策略主要是为了 **减少重复请求、提高加载速度、降低服务器压力**。  
整体来说 HTTP 缓存主要分为 **两种：强缓存和协商缓存**。

浏览器在请求资源时，一般会 **先判断强缓存，如果没命中，再走协商缓存**。

### 强缓存

强缓存的特点是：

**浏览器直接使用本地缓存，不会向服务器发送请求。**
也就是说，如果资源还在缓存有效期内，浏览器就直接从本地读取资源。

强缓存主要通过两个响应头控制：

**Cache-Control（HTTP/1.1）**

常见字段：
- `max-age=3600`：缓存 3600 秒
- `no-cache`：需要走协商缓存验证
- `no-store`：完全不缓存
- `public / private`：控制是否可以被共享缓存

例如：Cache-Control: max-age=3600

**Expires（HTTP/1.0）**
- 表示一个具体过期时间，例如：Expires: Wed, 21 Oct 2026 07:28:00 GMT

不过现在一般 **优先使用 Cache-Control**。

### 协商缓存

如果 **强缓存没有命中（比如过期了）**，浏览器就会发请求到服务器，询问资源是否变化，这就是 **协商缓存**。

如果资源没变，服务器会返回：304 Not Modified，浏览器继续使用本地缓存。

**Last-Modified / If-Modified-Since**

- 服务器返回资源最后修改时间：`Last-Modified: Tue, 01 Jan 2025 00:00:00 GMT`
- 下一次请求时浏览器会带上：`If-Modified-Since: Tue, 01 Jan 2025 00:00:00 GMT`

- `If-Modified-Since` **不是请求时间**，而是 **浏览器上次拿到资源时服务器返回的 Last-Modified 时间**。也就是说：**它代表“浏览器当前缓存的资源版本时间”**，而不是“本次请求的时间”。

- 如果 **资源修改时间 ≤ If-Modified-Since**  
    说明资源没有变化，服务器返回 **304 Not Modified**
- 如果 **资源修改时间 > If-Modified-Since**  
    说明资源已经更新，服务器返回 **200 + 新资源**

**ETag / If-None-Match**

ETag 是资源的 **唯一标识**（根据内容生成）

服务器返回：
- `ETag`: "abc123"
- 浏览器下次请求带上：
	- `If-None-Match`: "abc123"
	- 服务器通过这个值判断资源是否改变。

通常 **ETag 比 Last-Modified 更准确**。

实际应用中
- **静态资源**（JS、CSS、图片）用强缓存 + 文件指纹（hash），设置很长的 max-age，文件内容变了 hash 就变，URL 变了就重新请求。
- **HTML** 用协商缓存或 no-cache，保证能拿到最新的入口文件。

## HTTPS


HTTPS = **HTTP + SSL/TLS 加密层**
可以防止中间人攻击、窃听、篡改。


那么简单总结下有区别

* HTTPS协议需要到 CA 申请证书，一般免费证书较少，因而需要一定费用
* HTTP是超文本传输协议，信息是明文传输，HTTPS 则是具有安全性的 SSL 加密传输协议
* HTTP和HTTPS使用的是完全不同的连接方式，用的端口也不一样，前者是80，后者是443
* HTTP的连接很简单，是无状态的，HTTPS协议是由 SSL+HTTP 协议构建的可进行加密传输、身份认证的网络协议，比HTTP协议安全


为了解决HTTP协议的这一缺陷，需要使用另一种协议：

超文本传输协议HTTPS，为了数据传输的安全，HTTPS在HTTP的基础上加入了 SSL/TLS
SSL/TLS 本质上是一个安全通信协议，TLS是升级版的 SSL

`https:// = http:// + 加密层`

**HTTPS = 非对称加密（握手时用） + 对称加密（传输时用）**。

1. **验证网站身份**（证书）
	- 服务器会把一个 **数字证书** 发给浏览器。  
	- 这个证书就像网站的 **身份证**，由可信的机构（CA，比如 DigiCert）签发。
	- 浏览器会检查证书是否有效
2. **安全交换一个会话密钥**（公钥加密）
	- 会话密钥，就是一个密码，而这个密码本身采用非对称加密
		- 浏览器生成一个随机数
	    - 用服务器的 **公钥** 加密后发给服务器
	    - 服务器用自己的 **私钥** 解密
3. **用这个密钥加密所有通信**（对称加密）
	- 之后浏览器和服务器通信时：
		- 用刚刚协商好的 **会话密钥**
		- 采用 **对称加密算法（如 AES）**
		- `明文 → 加密 → 发送 → 解密 → 明文`
