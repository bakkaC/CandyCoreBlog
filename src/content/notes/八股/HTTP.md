## HTTP 1.1 特点

**灵活易用易于扩展**：

HTTP 协议里的各类请求方法、URI/URL、状态码、头字段等每个组成要求都没有被固定死，都允许开发人员自定义和扩充。

同时 HTTP 由于是工作在应用层，则它下层可以随意变化
- HTTP 在应用层
- 应用层只定义“数据格式/语义”（请求、响应）
- 不负责“传输细节”
- 传输细节由下层负责（TCP/UDP/QUIC）

比如：
- HTTPS 就是在 HTTP 与 TCP 层之间增加了 SSL/TLS 安全传输层；
- HTTP/1.1和 HTTP/2.0传输协议使用的是 TCP 协议，而到了 HTTP/3.0传输协议改用了 UDP 协议。

**无状态：**

无状态的好处，因为服务器不会去记忆 HTTP 的状态，所以不需要额外的资源来记录状态信息，这能减轻服务器的负担，能够把更多的 CPU 和内存用来对外提供服务。

无状态的坏处，既然服务器没有记忆能力，它在完成有关联性的操作时会非常麻烦。
例如登录->添加购物车->下单->结算->支付，这系列操作都要知道用户的身份才行。但服务器不知道这些请求是有关联的，每次都要问一遍身份信息。（可以用客户端保存身份信息状态解决）

**不安全**：使用明文传输，信息可能被获取，伪装，篡改——使用 HTTPS


## HTTP版本


HTTP 发展主要有 **HTTP/1.0、HTTP/1.1、HTTP/2 和 HTTP/3**。

**HTTP/1.0（短连接）**

- 每个请求都要建立一次 TCP 连接
- 请求完就断开（开销大）  
    问题：**效率低，延迟高**

**HTTP/1.1（长连接 + 管道）**

- 支持 **Keep-Alive**（复用 TCP 连接）
- 支持 **管道化（pipelining）**（但基本没用起来），可以一次发送多个请求
- 引入缓存、Host 头等优化

仍然有问题：
- **队头阻塞（Head-of-Line Blocking）**
- 一个连接同一时间基本只能按顺序处理

**3️⃣ HTTP/2（多路复用 + 二进制）**

- **二进制分帧**（更高效）
- **多路复用**（一个连接并发多个请求）
- **Header 压缩（HPACK）**
- 支持服务器推送

解决：
- 减少连接数
- 提高并发能力

但仍有问题：基于 TCP → **TCP 层仍有队头阻塞**

**4️⃣ HTTP/3（基于 QUIC / UDP）**

- 改用 **QUIC（UDP）** 替代 TCP
- 内置 TLS（更快建立连接）
- 真正解决 **队头阻塞问题**
- 支持连接迁移（网络切换不断线）




## HTTP 报文结构

HTTP 报文主要分成 **请求报文** 和 **响应报文**，两者的结构其实很类似，基本都由三部分组成：**起始行、头部字段、消息体**。

先说 **HTTP 请求报文**。

请求报文一般包含四部分：**请求行、请求头、空行、请求体**。

请求行主要包含三个信息：**请求方法、请求的 URL、HTTP 版本**。  
比如：

GET /index.html HTTP/1.1

接下来是 **请求头**，也就是一些键值对，用来描述客户端的信息，比如：

Host: www.example.com  
User-Agent: Chrome  
Accept: text/html  
Cookie: xxx

请求头后面会有一个 **空行**，这个空行的作用是用来分隔 **头部和消息体**。

最后是 **请求体**，也就是请求的数据，一般在 **POST、PUT** 这种请求里会有，比如提交表单或者 JSON 数据。

---

然后是 **HTTP 响应报文**。

响应报文结构也差不多，包括：**状态行、响应头、空行、响应体**。

**状态行**包含三个信息：**HTTP版本、状态码、状态码描述**。  
比如：

HTTP/1.1 200 OK

接下来是 **响应头**，用于描述服务器返回的信息，例如：

Content-Type: text/html  
Content-Length: 1024  
Set-Cookie: sessionid=xxx

然后同样会有一个 **空行**，用于分隔头部和内容。

最后是 **响应体**，也就是服务器返回的具体数据，比如 HTML 页面、JSON 数据、图片等。

## GET/POST和状态码

| 方法          | 作用              |
| ----------- | --------------- |
| **GET**     | 获取资源（最常见）       |
| **POST**    | 提交数据（如表单）       |
| **PUT**     | 替换资源            |
| **PATCH**   | 部分更新资源          |
| **DELETE**  | 删除资源            |
| **HEAD**    | 只请求头，不取内容（用于探测） |
| **OPTIONS** | 查询服务器支持哪些方法     |

1. 语义上 GET 用来查数据，POST 用于修改数据
2. 语义上 GET 请求是幂等的，多次调用不会有副作用，POST请求是非幂等的
3. GET 请求的参数位置一般是写在 URL 中，浏览器对 URL 有长度限制，POST 理论上没限制，但服务端通常会加限制
4. GET 会被浏览器和 CDN 缓存，POST不会

两者都不安全，安全性依赖 HTTPS，GET 也可以传请求体，用来改数据，POST 也可以用 URL 传参，用来查数据

全用 POST 可以把约定请求数据方式，减小沟通成本，
但是基建的缓存和重试一些策略会失效，GET 请求 URL和参数比较直观，POST 请求还需要构造请求体无形之中会增加排查问题的成本

| 状态码                           | 含义    |
| ----------------------------- | ----- |
| **200 OK**                    | 请求成功  |
| **301 Moved Permanently**     | 永久重定向 |
| **302 Found**                 | 临时重定向 |
| **304 Not Modified**          | 使用缓存  |
| **400 Bad Request**           | 请求错误  |
| **401 Unauthorized**          | 需要认证  |
| **403 Forbidden**             | 无权限访问 |
| **404 Not Found**             | 找不到资源 |
| **500 Internal Server Error** | 服务器错误 |
| **502 Bad Gateway**           | 网关错误  |
| **504 Gateway Timeout**       | 网关超时  |

## HTTP缓存

HTTP 的缓存策略主要是为了 **减少重复请求、提高加载速度、降低服务器压力**。  
整体来说 HTTP 缓存主要分为 **两种：强缓存和协商缓存**。

浏览器在请求资源时，一般会 **先判断强缓存，如果没命中，再走协商缓存**。

### 强缓存

强缓存的特点是：

**浏览器直接使用本地缓存，不会向服务器发送请求。**
也就是说，如果资源还在缓存有效期内，浏览器就直接从本地读取资源。

强缓存主要通过两个响应头控制：

**Cache-Control（HTTP/1.1）**

常见字段：
- `max-age=3600`：缓存 3600 秒
- `no-cache`：需要走协商缓存验证
- `no-store`：完全不缓存
- `public / private`：控制是否可以被共享缓存

例如：Cache-Control: max-age=3600

**Expires（HTTP/1.0）**
- 表示一个具体过期时间，例如：Expires: Wed, 21 Oct 2026 07:28:00 GMT

不过现在一般 **优先使用 Cache-Control**。

### 协商缓存

如果 **强缓存没有命中（比如过期了）**，浏览器就会发请求到服务器，询问资源是否变化，这就是 **协商缓存**。

如果资源没变，服务器会返回：304 Not Modified，浏览器继续使用本地缓存。

**Last-Modified / If-Modified-Since**

- 服务器返回资源最后修改时间：`Last-Modified: Tue, 01 Jan 2025 00:00:00 GMT`
- 下一次请求时浏览器会带上：`If-Modified-Since: Tue, 01 Jan 2025 00:00:00 GMT`

- `If-Modified-Since` **不是请求时间**，而是 **浏览器上次拿到资源时服务器返回的 Last-Modified 时间**。也就是说：**它代表“浏览器当前缓存的资源版本时间”**，而不是“本次请求的时间”。

- 如果 **资源修改时间 ≤ If-Modified-Since**  
    说明资源没有变化，服务器返回 **304 Not Modified**
- 如果 **资源修改时间 > If-Modified-Since**  
    说明资源已经更新，服务器返回 **200 + 新资源**

**ETag / If-None-Match**

ETag 是资源的 **唯一标识**（根据内容生成）

服务器返回：
- `ETag`: "abc123"
- 浏览器下次请求带上：
	- `If-None-Match`: "abc123"
	- 服务器通过这个值判断资源是否改变。

通常 **ETag 比 Last-Modified 更准确**。

实际应用中
- **静态资源**（JS、CSS、图片）用强缓存 + 文件指纹（hash），设置很长的 max-age，文件内容变了 hash 就变，URL 变了就重新请求。
- **HTML** 用协商缓存或 no-cache，保证能拿到最新的入口文件。

## HTTPS


HTTPS = **HTTP + SSL/TLS 加密层**
可以防止中间人攻击、窃听、篡改。


那么简单总结下有区别

* HTTPS协议需要到 CA 申请证书，一般免费证书较少，因而需要一定费用
* HTTP是超文本传输协议，信息是明文传输，HTTPS 则是具有安全性的 SSL 加密传输协议
* HTTP和HTTPS使用的是完全不同的连接方式，用的端口也不一样，前者是80，后者是443
* HTTP的连接很简单，是无状态的，HTTPS协议是由 SSL+HTTP 协议构建的可进行加密传输、身份认证的网络协议，比HTTP协议安全


为了解决HTTP协议的这一缺陷，需要使用另一种协议：

超文本传输协议HTTPS，为了数据传输的安全，HTTPS在HTTP的基础上加入了 SSL/TLS
SSL/TLS 本质上是一个安全通信协议，TLS是升级版的 SSL

`https:// = http:// + 加密层`

**HTTPS = 非对称加密（握手时用） + 对称加密（传输时用）**。

对称加密：加密和解密用相同的密钥，
- 无法直接传输，因为双方都要有这个密钥才能通信
- 密钥的传输过程不安全，双方都要有意味着密钥必须经过传输
- 经过传输的密钥就会被拦截，导致消息传输不安全

非对称加密：一把公钥，一把私钥，加密解密使用不同的方式

- 公钥加密私钥解密，可以实现消息的保密传输
	- 前提：双方都有各自的公钥和私钥
	- A=>B传输消息
		- 让 B 把公钥 B传给 A
		- A 把消息用公钥 B进行加密，而这个消息只有私钥 B才能解密
		- 即使拿到了传输过程中的公钥 B也无法破解消息内容
	- B=> A则是相反使用 A的公钥和私钥
- 所以这就实现了消息的保密传输

那这不是可以用来解决上述对称加密的痛点吗，可以用非对称加密传输对称密钥，这样就能防止消息被看到了

中间人攻击
- 这里还有问题，虽然非对称加密的公钥不怕被看到，但是怕被篡改，这会导致对称加密的泄露
	- 非对称加密的第一步是让接收方给 发送方 接收方的公钥
	- 中间人在这时拦截公钥，伪造一个fake公钥，传给发送方，那么发送就会用假公钥加密信息
	- 那发消息的时候，中间人就可以用对应fake 私钥解密，拿到对称密钥了
	- 这时再用正确公钥重新加密，就可以骗过接收方，完美犯罪
- 这就是中间人攻击

那么如何防篡改呢：
- 私钥加密公钥解密
	- A=>B传输消息
		- A 给 B传输公钥 A，用私钥 A 加密信息，发送
		- 中间人用相同的手法拿到公钥 A，解密拿到信息
		- 但没有私钥无法重新加密，实现完美犯罪，虽然消息泄露，但是无法伪造发送接收方

那怎么又防止篡改，又防止偷看呢
核心问题是公钥传输无法安全，这怎么解决呢，解决办法是想办法识别公钥是否被篡改

答案是通过 CA 机构，即一个可信的第三方，假设你要给对方自己的公钥：
- 生产的设备会自带可信的CA 公钥，因此访问 CA 不需要经过公钥传输
- 这使得和 CA 的通信是安全的，这是前提
- 那么可以直接把公钥给 CA，让 CA 进行签名（私钥加密，公钥解密），发给自己
- 然后同时把公钥和签名（ CA 加密过的身份标识 ），集合起来就是一个证书，发给对方
- 由于对方可以使用内置的 CA 公钥，解密签名，拿到证书，通过验证证书可以确定公钥是否可信


这整套就是 TLS：
- 证书传输（公钥）
- CA 验证（防中间人）
- 密钥交换（非对称）
- 后续通信（对称加密）


1. **验证网站身份**（证书）
	- 服务器会把一个 **数字证书** 发给浏览器。  
	- 这个证书就像网站的 **身份证**，由可信的机构（CA，比如 DigiCert）签发。
	- 浏览器会检查证书是否有效
2. **安全交换一个会话密钥**（公钥加密）
	- 会话密钥，就是一个密码，而这个密码本身采用非对称加密
		- 浏览器生成一个随机数
	    - 用服务器的 **公钥** 加密后发给服务器
	    - 服务器用自己的 **私钥** 解密
3. **用这个密钥加密所有通信**（对称加密）
	- 之后浏览器和服务器通信时：
		- 用刚刚协商好的 **会话密钥**
		- 采用 **对称加密算法（如 AES）**
		- `明文 → 加密 → 发送 → 解密 → 明文`
