# **一、基本做法：标记清理（Mark-Sweep）**

传统 GC 流程：

1. 开始时认为所有对象都是“需要清理”的（白色）。
2. 从根对象出发遍历，把访问到的对象标记为 active（黑色）。
3. 遍历结束后，未标记的对象即为垃圾。
4. 执行清理。

---

# **二、V8 的代际优化（Generational GC）**

V8 把堆划分成：

- **新生代（新对象）**
- **老生代（存活较久的对象）**


不同区域使用不同的回收算法，以获得最优性能。

---

# **三、新生代：Scavenge（复制算法 Copying GC）**

### **1. 新生代特点**

- 对象生命周期短，大部分对象在创建后很快死亡。
- 空间较小（几 MB）。
- 必须高频快速回收。

### **2. Scavenge 机制（From → To 复制）**

新生代堆分成两个等大小的半区：

- From 区（使用中）
- To 区（空闲）

GC 流程：

1. 扫描 From 区的对象。
2. 将存活对象（active）复制到 To 区。
3. 执行引用更新。
4. From/To 角色互换。

### **3. 复制算法的优点**

- **只处理活对象，死对象自动被淘汰，不需要额外清理。**
- **没有内存碎片**（To 区是连续写入）。
- **分配速度快**（bump pointer 方式分配）。
- 非常适合新生代“死亡率高”的特性。

### **4. 晋升到老生代**

对象如果：

- 经历多次新生代 GC 仍然存活，或
- 占用空间较大

则会被晋升到老生代。

---

# **四、老生代：标记 + 清理 + 整理（Mark-Sweep + Mark-Compact）**

老生代对象多、存活时间长，不能用复制算法（成本太高）。

流程：

1. **标记（Mark）**  
    从根对象扫描，标记可达对象。

2. **清理（Sweep）**  
    回收未标记对象，但可能会产生碎片。

3. **整理（Compaction）**  
    将存活对象移动，避免碎片（仅在需要时做，不是每次都做）。


---

# **五、V8 老生代优化：三色标记 + 增量标记**

老生代空间大，不能长时间阻塞 JS 线程，因此使用**增量标记**（Incremental Marking）：

- 把原本一次性做完的“标记”拆分，让 JS 和 GC 交替执行。

- 每次只处理一小部分标记任务。


### **三色法：**

- **白色**：未访问（可能是垃圾）

- **灰色**：已访问，子节点未处理

- **黑色**：已访问，子节点也已处理


GC 持续把灰色处理成黑色，直到无灰色对象。

---

# **六、为什么需要写屏障（Write Barrier）**

在增量标记过程中 JS 仍在执行，可能会出现：

- 已标黑的对象新增指向白对象的引用

- 这会破坏三色不变式，造成漏标


解决方案就是写屏障：

### **写屏障的作用**

当 JS 代码执行“黑 → 白”引用时，自动把白对象变成灰色，加入标记队列。

保证：

- 所有可达对象最终都会被标记到（变黑）

- 清扫时不会误删仍然可达的对象


---

# **七、增量标记为什么安全？**

- 增量标记阶段**不执行清扫**，不会删除对象。

- 最终会有一个短暂的 **Final Mark（最终标记）**：

    - 补齐所有可能漏掉的标记

    - 处理灰色队列

- Final Mark 之后，所有可达对象都为黑色；  
    白色对象才真正不可达，可执行清扫。


---

# **八、整体流程总结**

### **新生代**

- 算法：Scavenge（复制）

- 特点：速度极快、无碎片、适用于大量短命对象

- 晋升：多次存活的对象进入老生代


### **老生代**

- 算法：标记-清除 + 必要时压缩

- 优化：三色法、增量标记、写屏障

- 目标：减少 STW 时间（不阻塞 JS）
